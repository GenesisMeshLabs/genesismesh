# v0.38.0 Plan -- Cascade-Resilient Consensus

## Positioning

v0.36 introduced K-of-N distributed consensus authorization.  The threshold
mechanism is sound: no single validator can approve a high-stakes action alone.
But the current design has a structural gap: it counts *votes*, not *independent
votes*.  If an adversary can cause multiple validators to derive their approval
from the same corrupted reasoning chain -- a persuasion cascade -- the K-of-N
threshold is met while the independence invariant is broken.

arXiv:2603.15809 ("Don't Trust Stubborn Neighbors") demonstrates this precisely:
"a single adversary can trigger a persuasion cascade that reshapes group
decisions."  The Friedkin-Johnsen model shows that when agents share context
before voting, the effective number of independent opinions collapses to one.
The defense is trust-adaptive weighting that *discounts correlated inputs*.

v0.38 closes this gap by adding vote-independence scoring to `ConsensusProof`
assembly.  Validators now include a `context_digest` (hash of their independent
local reasoning state at vote time) and a `submitted_at` timestamp.  The
assembler computes a `CascadeScore` from the distribution of context digests
and the time-correlation of approvals.  If the score exceeds the threshold,
assembly is blocked.

> **Scope constraint**: This is an extension of the existing ConsensusProof
> assembly path.  It does not change the verification protocol for a
> pre-assembled ConsensusProof.  Out of scope: cryptographic proof of
> independence, MPC-style blind voting, and any change to the BoundaryEngine
> normal path.

v0.38 should prove:

> A `ConsensusProof` assembled when CascadeScore > threshold cannot be produced
> even if K validators approve, if those K validators submitted context_digests
> that are statistically indistinguishable from a single information source.
> `verify_consensus_proof()` returns `cascade_detected` when the assembled proof
> carries a cascade score above the allowed threshold.

## Why this is next

The triage recommendation in the 2026 research analysis is explicit: "Persuasion
Cascades (Topic #2) and Adversarial Seed Isolation (Topic #5) represent the
highest risk to the current release."  Cascade-Resilient Consensus targets the
v0.36 artifact directly.  Adversarial Seed Isolation (v0.39) targets v0.37.
These two plans together close the most urgent attack surface opened by the
Phase I completion.

## Design: vote independence scoring

Two signals measure independence:

**Context divergence score (CDS)**
Each `ValidatorVote` carries a `context_digest`: SHA-256 of the tuple
`(justification_proof_digest, local_risk_signal_digest, validator_state_nonce)`.
The `validator_state_nonce` is a random value generated by the validator at
vote time.  The assembler collects the set of unique context digests across
approve votes.  CDS = fraction of approve votes sharing the modal context
digest.  CDS = 1.0 means all approvers saw identical state.

**Temporal clustering score (TCS)**
TCS measures whether approve votes arrived in an unusually tight time window
(suggesting a broadcast trigger rather than independent deliberation).
TCS = 1.0 - (std_dev(vote_arrival_times) / expected_deliberation_seconds).
TCS = 1.0 means all votes arrived simultaneously.

**Cascade score**
CascadeScore = (cds_weight * CDS) + (tcs_weight * TCS),
where defaults are cds_weight=0.7, tcs_weight=0.3.

A `ConsensusProof` is blocked if CascadeScore > cascade_threshold (default 0.4).

class CascadeAssessment(BaseModel):
    assessment_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    consensus_id: str
    cascade_score: float = Field(..., ge=0.0, le=1.0)
    context_divergence_score: float = Field(..., ge=0.0, le=1.0)

    unique_context_count: int

    blocked: bool
    threshold_used: float

### Test plan: `genesis_mesh/tests/test_cascade_resilient_consensus.py`

~28 tests:
- `assess_cascade_risk()`: all unique context digests -> independent
- All identical context digests -> cascade_detected
- Majority identical, minority unique -> cascade_detected
- Tight temporal window -> high TCS
- Spread temporal window -> low TCS
- Configurable weights (cds_weight=1.0 disables TCS)
- Cascade threshold 0.0 disables checking (always independent)
- `assemble_consensus_proof()`: blocks when cascade detected
- `assemble_consensus_proof()`: succeeds when independent
- `verify_consensus_proof()`: returns `cascade_detected` for tampered proof
- ValidatorVote requires context_digest
- CLI: `assess-cascade` exit 0 on independent, 1 on cascade
- Backward compatibility: votes without context_digest fail with `missing_context_digest`
